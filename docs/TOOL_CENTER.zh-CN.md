# 工具中心使用文档（新手版）

> 本文把用户当成第一次接触 ops-mcp 的用户来讲解。你不需要懂 Go、React、Kubernetes、Prometheus 或 MCP，也可以按本文完成工具查看、执行、新增、编辑、删除、审批和审计追踪。

## 1. 工具中心是什么？

工具中心是 ops-mcp 里的“运维工具入口”。你可以把它理解为一个受管控的工具箱：

- 每一行都是一个可以执行的工具，例如查看磁盘、查看内存、查询 Kubernetes Pod、查询 Prometheus 指标。
- 每个工具都有说明、分类、风险等级、是否只读、是否需要审批、需要填写哪些参数。
- 你执行工具时，系统不会“偷偷执行”。它会先检查权限和风险，再决定：
  - 直接执行；
  - 提交审批；
  - 拒绝执行。
- 每次操作都会写入执行记录和审计记录，方便之后追踪“谁在什么时候对什么目标做了什么”。

默认 mock 模式是安全演示模式：系统返回模拟数据，不连接真实生产环境，也不会修改真实机器或集群。

## 2. 如何打开工具中心？

1. 打开浏览器。
2. 进入前端地址：

```text
http://localhost:5173
```

如果是在部署服务器上访问，请使用部署服务器地址，例如：

```text
http://192.168.20.166:5173
```

3. 看左侧菜单。
4. 点击 **工具中心**。

进入后，你会看到一个工具列表表格。

## 3. 工具列表每一列是什么意思？

工具中心表格通常包含这些列：

- **工具**：工具的唯一名称，例如 `linux.disk_usage`。执行或调用 API 时都用这个名称。
- **分类**：工具属于哪一类，例如：
  - `linux`：Linux 主机检查工具；
  - `kubernetes`：Kubernetes 查询工具；
  - `prometheus`：Prometheus 指标查询工具；
  - `custom`：用户自己新增的工具。
- **风险**：系统对工具风险的标记：
  - `低` / `low`：一般只读查询，通常可以直接执行；
  - `中` / `medium`：可能包含较敏感信息或影响范围较大，默认会进入审批；
  - `高` / `high`：高风险操作，默认也会进入审批；
  - `严重` / `critical`：默认拒绝执行。
- **类型**：
  - `只读`：理论上只查看信息，不修改系统；
  - `变更`：可能修改系统状态。
- **说明**：用一句话解释工具用途。
- **操作**：你可以点击：
  - **详情**：查看工具详细信息和输入参数说明；
  - **编辑**：修改工具的元数据；
  - **执行**：填写参数并运行工具；
  - **删除**：删除工具。

## 4. 当前默认有哪些常用工具？

### 4.1 Linux 工具

- `linux.system_info`：查看主机名、内核、发行版、架构、运行时长和虚拟化信息。
- `linux.load_average`：查看 1/5/15 分钟负载均值和 CPU 核心数。
- `linux.memory_usage`：查看内存和 swap 使用情况。
- `linux.disk_usage`：按路径查看磁盘使用情况。
- `linux.process_list`：查看进程列表以及 CPU/内存占用。
- `linux.network_interfaces`：查看网卡状态、IP 地址和流量计数。
- `linux.service_status`：查看 systemd 服务状态。
- `linux.journal_tail`：查看 systemd journal 日志尾部；这是中风险工具，需要审批。
- `linux.ping`：检查网络连通性。
- `linux.dns_lookup`：检查 DNS 解析结果。

### 4.2 Kubernetes 工具

- `k8s.list_pods`：列出 Pod。
- `k8s.get_pod_logs`：查看 Pod 日志。
- `k8s.list_events`：列出事件。
- `k8s.get_deployment_status`：查看 Deployment 状态。

### 4.3 Prometheus 工具

- `prometheus.query`：执行只读 PromQL 查询。
- `prometheus.service_error_rate`：查看服务错误率。
- `prometheus.service_latency_p95`：查看服务 P95 延迟。
- `prometheus.pod_cpu_usage`：查看 Pod CPU 使用量。
- `prometheus.pod_memory_usage`：查看 Pod 内存使用量。

## 5. 如何搜索和筛选工具？

工具很多时，可以使用顶部筛选区。

### 5.1 搜索框

在“搜索工具/说明”里输入关键字，例如：

- 输入 `linux`：只看 Linux 工具。
- 输入 `disk`：找到磁盘工具。
- 输入 `pod`：找到 Kubernetes Pod 相关工具。
- 输入 `dns`：找到 DNS 查询工具。

搜索会匹配工具名称、说明和分类。

### 5.2 分类筛选

点击“全部分类”，可以只看某一类工具：

- `linux`
- `kubernetes`
- `prometheus`
- `custom`

### 5.3 风险筛选

点击“全部风险”，可以按风险等级筛选：

- 低
- 中
- 高
- 严重

如果你只想找能尽量直接执行的查询工具，通常先选“低”。

### 5.4 类型筛选

点击“全部类型”，可以选择：

- 只读
- 变更

新手建议优先使用“只读”工具。

## 6. 如何查看工具详情？

1. 在工具列表中找到一个工具，例如 `linux.disk_usage`。
2. 点击该行右侧的 **详情**。
3. 右侧会打开详情抽屉。
4. 你可以看到：
   - 工具名称；
   - 风险等级；
   - 是否只读；
   - 工具说明；
   - 分类；
   - 是否需要审批；
   - 输入 Schema。

### 6.1 什么是输入 Schema？

输入 Schema 是工具需要你填写的参数说明。

例如：

```json
{
  "path": "string?"
}
```

意思是：

- 参数名是 `path`；
- 参数类型是字符串；
- 后面的 `?` 表示这个参数可填可不填。

再例如：

```json
{
  "host": "string",
  "count": "number?"
}
```

意思是：

- `host` 是必填字符串；
- `count` 是可选数字。

常见写法：

- `string`：必填文本，例如 `example.com`。
- `string?`：可选文本。
- `number`：必填数字，例如 `10`。
- `number?`：可选数字。

## 7. 如何执行工具？

下面以 `linux.disk_usage` 为例，演示一次完整执行。

### 7.1 打开执行弹窗

1. 找到 `linux.disk_usage`。
2. 点击 **执行**。
3. 页面会打开“执行工具：linux.disk_usage”的弹窗。

### 7.2 执行弹窗里的字段是什么意思？

弹窗里有几个字段：

- **执行人**：谁在执行这个工具。默认可能是 `admin`。
- **角色**：执行人的角色：
  - `viewer`：查看者，只适合执行只读工具；
  - `operator`：运维人员，可以执行更多工具；
  - `admin`：管理员，权限最高。
- **目标**：这次操作的目标，方便审计。例如：
  - `local-dev`
  - `host=demo`
  - `cluster=demo namespace=default`
- **授权**：
  - `未授权`：正常执行请求，遇到中/高风险工具通常会进入审批；
  - `已授权`：表示你已经拿到授权，系统会按已授权请求处理。
- **参数 JSON**：工具真正需要的参数，必须是合法 JSON 对象。

### 7.3 JSON 参数怎么写？

`linux.disk_usage` 的参数可以这样写：

```json
{
  "path": "/var"
}
```

注意：

- JSON 必须用英文双引号 `"`，不能用中文引号。
- 最外层必须是 `{}`。
- 冒号后面的字符串也要加双引号。
- 最后一行不要多写逗号。

错误示例：

```json
{
  path: /var,
}
```

正确示例：

```json
{
  "path": "/var"
}
```

### 7.4 提交执行

1. 填好参数后点击 **执行**。
2. 如果成功，弹窗下方会显示执行结果。
3. 对于 `linux.disk_usage`，你会看到类似：

```json
{
  "status": "completed",
  "message": "completed",
  "data": {
    "path": "/var",
    "filesystem": "/dev/vda1",
    "totalGiB": 120,
    "usedGiB": 42.7,
    "availableGiB": 77.3,
    "usedPercent": 35.6
  }
}
```

这表示工具执行完成，并返回了磁盘容量和使用率。

## 8. 为什么有些工具执行后不是完成，而是“已提交审批”？

如果工具风险较高，或者工具被设置为“需要审批”，系统不会马上执行，而是创建一条审批请求。

例如 `linux.journal_tail`：

- 风险：中；
- 需要审批：是；
- 因此第一次执行时通常返回：`pending_approval`。

你会看到类似结果：

```json
{
  "status": "pending_approval",
  "message": "pending approval",
  "approvalId": "..."
}
```

这表示：

- 请求已经记录；
- 工具还没有真正返回日志；
- 需要到“审批中心”处理。

## 9. 如何处理审批？

1. 点击左侧菜单的 **审批中心**。
2. 找到刚才创建的审批请求。
3. 查看工具名、执行人、目标、原因。
4. 如果确认可以执行，点击 **批准**。
5. 如果不允许执行，点击 **拒绝**。

注意：当前 MVP 里的审批中心主要记录审批决定。对于真实生产系统，批准后通常还需要配合“重新执行”或后端任务调度机制完成最终动作；具体行为以后端实现为准。

## 10. 如何查看执行历史？

1. 点击左侧菜单的 **执行中心**。
2. 你会看到每次工具执行的记录。
3. 常见状态包括：
   - `completed`：已完成；
   - `pending_approval`：等待审批；
   - `denied`：被策略拒绝；
   - `validation_failed`：参数校验失败；
   - `error`：执行过程出错。
4. 点击记录可以查看：
   - 输入参数；
   - 返回结果；
   - 执行人；
   - 目标；
   - 审计 ID。

## 11. 如何查看审计记录？

1. 点击左侧菜单的 **审计中心**。
2. 你可以看到谁执行了什么工具，以及系统是否允许。
3. 审计记录通常包括：
   - 执行时间；
   - 执行人；
   - 角色；
   - 动作，例如 `tool.linux.disk_usage`；
   - 目标；
   - 是否允许；
   - 原因；
   - 参数。

敏感参数名会自动脱敏，例如：

- `password`
- `secret`
- `token`
- `api_key`
- `authorization`

所以不要把密钥写进普通说明里；如果确实传入敏感字段，系统会尽量在审计记录中隐藏值。

## 12. 如何新增一个工具？

工具中心支持新增自定义工具。

1. 点击右上角 **新增工具**。
2. 填写表单。

字段说明：

- **工具名称**：必须唯一，不能有空格或斜杠。例如：
  - `custom.echo`
  - `linux.check_port`
- **分类**：例如 `custom`、`linux`、`database`。
- **说明**：让用户看懂这个工具做什么。
- **风险**：低、中、高、严重。
- **只读工具**：
  - 打开：表示工具只查询，不修改；
  - 关闭：表示工具可能修改系统。
- **需要审批**：
  - 打开：每次执行前会进入审批；
  - 关闭：是否审批由风险和策略决定。
- **输入 Schema(JSON 对象)**：描述参数。

示例：新增一个 echo 工具：

```json
{
  "message": "string"
}
```

保存后，工具会出现在列表里。

重要说明：当前自定义工具使用默认 mock handler。也就是说，它会走完整的策略、审批、执行和审计流程，但不会真的调用外部系统；返回内容会包含你传入的参数和“custom tool executed”这类模拟结果。

## 13. 如何编辑工具？

1. 找到目标工具。
2. 点击 **编辑**。
3. 修改说明、分类、风险、是否只读、是否需要审批、输入 Schema。
4. 点击确认保存。

注意：

- 工具名称不能改。如果要改名，请删除后重新新增。
- 风险从低改成中/高后，后续执行可能会进入审批。
- 打开“需要审批”后，即使是低风险工具，也会进入审批。

## 14. 如何删除工具？

1. 找到目标工具。
2. 点击删除图标。
3. 在确认框中点击删除。

删除后：

- 工具不会再出现在工具中心。
- 旧的执行记录和审计记录仍可能保留，用于历史追踪。
- 如果删除的是默认工具，重启服务后可能会再次由后端默认注册，具体取决于后端启动逻辑。

## 15. 工具执行背后的逻辑是什么？

当你点击执行后，系统大致会做这些事情：

1. 找到工具定义。
2. 读取你填写的执行人、角色、目标、授权状态和参数。
3. 用 Policy Engine 做策略判断。
4. 如果策略拒绝：
   - 生成执行记录，状态为 `denied`；
   - 生成审计记录；
   - 返回错误。
5. 如果策略允许，但工具风险较高或被设置为需要审批：
   - 生成执行记录，状态为 `pending_approval`；
   - 生成审批请求；
   - 返回审批 ID。
6. 如果允许直接执行：
   - 调用工具 handler；
   - 保存返回结果；
   - 生成执行记录；
   - 生成审计记录；
   - 返回 `completed`。

## 16. 角色和权限怎么理解？

当前 MVP 的默认策略可以这样理解：

- `viewer`：适合查看信息。只读工具通常允许，写工具会被拒绝。
- `operator`：适合运维人员。只读工具允许，中风险写工具在开发/预发环境可能允许，生产写操作需要审批。
- `admin`：管理员，默认允许大部分操作。

但无论角色如何：

- `critical` 严重风险工具默认拒绝。
- 工具如果设置了 `requiresApproval=true`，即使风险是低，也会进入审批。
- 中/高风险工具默认会进入审批流程。

## 17. 风险等级怎么选择？

新增或编辑工具时，可以参考：

- **低 / low**：只读查询，不暴露敏感信息，不修改系统，例如查看磁盘使用率。
- **中 / medium**：可能包含敏感信息或影响较大，例如查看日志、查询较大范围的数据。
- **高 / high**：可能影响系统稳定性或涉及变更，例如重启服务、清理缓存。
- **严重 / critical**：危险操作，例如删除数据、删除资源、执行任意命令。当前默认拒绝。

不确定时，选择更高风险，并打开“需要审批”。

## 18. 新手推荐操作路径

如果你是第一次使用，建议按这个顺序体验：

1. 打开工具中心。
2. 在搜索框输入 `linux`。
3. 点击 `linux.system_info` 的 **详情**，看懂工具信息。
4. 点击 `linux.disk_usage` 的 **执行**。
5. 参数填写：

```json
{
  "path": "/"
}
```

6. 点击执行，查看返回结果。
7. 打开执行中心，查看刚才的执行记录。
8. 打开审计中心，查看刚才的审计记录。
9. 再执行 `linux.journal_tail`，观察它进入审批中心。
10. 打开审批中心，批准或拒绝这条请求。

## 19. 常见问题

### 19.1 点击执行后提示 JSON 错误怎么办？

检查：

- 是否用了英文双引号；
- 最外层是否是 `{}`；
- 最后一项后面是否多写了逗号；
- 是否把数字写成了字符串，或把字符串漏了引号。

### 19.2 为什么低风险工具也进入审批？

可能是工具打开了“需要审批”。工具配置里的 `requiresApproval=true` 优先级很高，即使风险是低，也会进入审批。

### 19.3 为什么工具被拒绝？

常见原因：

- 角色权限不足；
- 工具不是只读，但角色是 `viewer`；
- 工具风险是 `critical`；
- 生产环境变更没有审批。

### 19.4 删除工具后为什么又出现了？

默认 mock 工具是后端启动时注册的。如果删除默认工具后重启服务，它可能会重新出现。自定义工具是否持久化取决于后端存储实现。

### 19.5 当前工具会真的操作 Linux 服务器吗？

默认 mock 模式不会。当前 Linux 工具返回模拟数据，用来演示工具中心、审批、执行和审计链路。接入真实服务器时，需要后端 adapter 改为真实实现，并配置安全边界。

## 20. 使用建议

- 新手优先执行只读、低风险工具。
- 不确定参数时，先点“详情”查看输入 Schema。
- 执行前确认“目标”写清楚，方便之后审计。
- 涉及日志、敏感信息、生产环境或变更操作时，打开“需要审批”。
- 不要把真实密码、Token、私钥直接写在普通参数或说明里。
